SSO-only login (disable password login)

An opt-in mode (DisablePasswordLogin) can disable native username/password login so users must sign in through SSO. Read these honest limits before enabling it:

  • There is no global Jellyfin kill-switch. Jellyfin core has no server-wide "disable password login" setting, and this plugin cannot intercept its password endpoint. The mode is enforced per account by repointing each user's authentication provider away from the built-in password provider. It is therefore "SSO-only for every account except a designated break-glass admin", not an absolute password kill-switch.
  • A break-glass admin always keeps password login. You must designate one existing, enabled administrator account that still has a password. That account is never repointed, so it can always log in with a password even if the identity provider is unreachable. Keep its password strong and stored offline — it is the org's root of recovery. Enabling the mode is refused unless such an admin is provable.
  • The mode is reversible without SSO. Turning it off restores native password routing for the accounts the mode repointed and never resets or reveals any password.
  • Total-lockout recovery works from disk. If you are ever fully locked out, stop Jellyfin, edit the plugin's config.xml, set DisablePasswordLogin back to false, and restart. On startup the plugin reconciles the user database to the flag — the accounts the mode repointed are restored to the built-in password provider (password hashes untouched), so native login works again. Only accounts the mode itself moved are restored; accounts created by SSO keep their SSO-only routing.
  • No separation of duties. Jellyfin has a single admin role, so one administrator can both design and activate this mode; the plugin cannot manufacture dual control. The toggle and the break-glass designation are elevation-gated and written to the ordinary Jellyfin log (which is not a tamper-evident audit trail).

The mode, its break-glass designation, and the reversible off-switch are driven by the elevation-gated SSO-Only/Enable, SSO-Only/Disable, SSO-Only/BreakGlassAdmin, and SSO-Only/Status endpoints (they run the fail-closed last-admin guard and the per-user enforcement sweep). Saving this configuration page never changes the SSO-only state. See the plugin wiki's Provider Setup page for the full runbook.

Export downloads a redacted copy of this plugin's configuration (provider secrets and account links are never included). Import merges an exported file into this instance, keeping the stored secret and account links of each unchanged provider — but if the import gives an OpenID provider a different discovery endpoint or client id, that provider's links and secret are cleared (a repoint safety measure), so review a file before importing it over a live provider. Re-enter each provider's secret here after importing. Requires administrator privileges.

Hardening

Refuse login unless the provider's discovery document advertises PKCE (S256) in code_challenge_methods_supported (RFC 9700 §2.1.1). When off, an unsupported provider only logs an audit warning and the login proceeds.

OIDC prompt parameter, e.g. login to force re-authentication or consent. Leave blank to omit it.

Refuse any login whose verified id_token does not return an acr within Acr Values above — a fail-closed step-up / forced-MFA gate. Off by default. Requires Acr Values to be set (the save is rejected otherwise, so a mis-set cannot lock out a userbase). The break-glass password admin is unaffected.

Account adoption (sensitive)

⚠ Sensitive: lets a first SSO login adopt (take over) a pre-existing, unlinked Jellyfin account whose username matches the SSO name, instead of rejecting it. Off by default (fail closed). Enable only for the one-time legacy-link migration window described in the account-linking runbook on the wiki, then turn it back off. A name-matched administrator account is never adopted regardless of this setting, and each adoption is recorded as an audit event.
When the identity provider's audience is broader than your intended Jellyfin audience: an unknown identity's first successful SSO login creates its account disabled, with no permissions, and the login is refused with an "awaiting administrator approval" message instead of a session. Approve the account by enabling it in the Jellyfin dashboard (Administration → Users). Off by default (new accounts are created enabled). This never disables an existing or adopted account — only a brand-new one — and each deferred provisioning is recorded as an audit event.
⚠ Sensitive: only meaningful with Allow Adopting Existing Accounts above. Additionally requires the login to carry a provider-verified email (email_verified == true) before adopting a same-named account, narrowing who can exploit the name-based match. Needs the email scope so the provider actually returns email_verified; without it, every adoption is refused. Off by default so an existing deployment relying on adoption is not silently locked out. An administrator account is never adopted regardless of this setting.

A login whose email_verified is not exactly true (absent, false, or unparseable) is refused. Needs the email scope so the provider actually returns email_verified; without it, every login is refused. Insecure options

These options disable OpenID Connect security defenses. Enable one only if your provider genuinely requires it; each is recorded as an audit warning.
⚠ Insecure: drops TLS on the discovery, JWKS and token endpoints, so the id_token signing keys can be swapped by a man-in-the-middle and a forged login accepted. Use only on a trusted internal network. Enabling it is recorded as an audit warning.
⚠ Insecure: disables endpoint-origin validation (a mix-up defense). May be required for Google OpenID, whose endpoints live on different hosts. Enabling it is recorded as an audit warning.
⚠ Insecure: disables matching the discovery issuer, a mix-up defense. Only for a provider that legitimately presents a different issuer (some templated/multi-tenant setups). Enabling it is recorded as an audit warning.
⚠ Insecure: disables the RFC 9207 iss check, an OpenID Connect mix-up defense. Only for a provider whose authorization response legitimately carries a different issuer. Enabling it is recorded as an audit warning.
Validates the saved provider: the server fetches the OpenID discovery document over the same hardened, HTTPS-checked path the login uses and reports the issuer, endpoints and JWKS reachability. Save the provider first. Requires administrator privileges; the client secret is never shown.
Saving requires a restart of Jellyfin to take effect.
